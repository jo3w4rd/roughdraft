Samples

The Data Oriented Tech Stack Sample repository includes basic examples demonstrating how to use DOTS in a Unity Project.

HelloECS

This is a set of projects that demonstrates different approaches with the MonoBehaviour, Hybrid Entity Component System and Pure Entity Component System. This is a good starting point to understand how the Entity Component System paradigm works. This is a set of projects demonstrate the absolute basics of the Unity ECS architecture.

HelloCube_01_ForEach

This example creates a single, rotating cube. The example demonstrates the separation of data and behavior in ECS using Systems and Components, as well as how to define ECS Entities using normal GameObjects.

The ComponentSystem used to update the Cube's rotation every frame still executes on the main thread, so this example does not take advantage of the available CPU cores. The ComponentSystem calls your ForEach method for every Entity that has the required Components specified by the method.

Systems implemented with ComponentSystem should generally only be used for tasks that must be performed on the main thread, but can also be used a stepping stone toward converting your existing game behaviors to the Entity-Component-System architecture.

HelloCube_02_IJobProcessComponentData

This example also creates a rotating cube. It uses the same Components as HelloCube_01_ForEach, but implements the Cube rotation with a Job-based System. The IJobProcessComponentData interface used in this example executes the job for each Entity that satisfies a System's requirements.

Systems implemented with IJobProcessComponentData are the recommended approach and can take advantage of available CPU cores.

HelloCube_03_IJobChunk

This example shows how to write a System using IJobChunk instead of IJobProcessComponentData. The IJobChunk interface executes a job for each block of memory, called a Chunk, that contains Entities which satisfy a System's requirements. (The Entities in a given Chunk all have the same Components.) Your job can then iterate over the Entities in each Chunk.

IJobChunk is the recommended approach to processing Components for cases more complex than a simple IJobProcessComponentData can describe.

HelloCube_04_Spawner

This example demonstrates how to spawn multiple Entities at runtime. The example uses a ComponentGroup to get the Entities of all spawners placed in the World. The example also defines an EntityArchetype, which it uses when it creates the spawned Entities.

Note that creating and destroying Entities must be carried out on the main thread, so the CubeSpawnerSystem in the example is implemented as a ComponentSystem.

Boids

The Boids example provides a more complex scenario with thousands of Entities. Boids simulates an underwater scene with a shark and schools containing thousands of fish. (It uses the classic Boids flocking algorithm for the schooling fish behavior.)

The Boids implements the following Systems:

  • SpawnRandomInSphereSystem — spawns a school of fish within a sphere using the Prefab objects, "TestBoidFish" and "TestBoidFish 1".
  • RandomInitialHeadingSystem — assigns random headings to the newly spawned fish.
  • BoidSystem — calculates the desired heading for each fish and updates their LocalToWorld transform matrix every frame.

The example controls the movement of the shark and the two target fish using the Unity animation system. The position of these GameObjects is copied to the ECS Position Component every frame by the CopyTransformFromGameObjectSystem. (Note that copying data from GameObject MonoBehaviours is inefficient and should, ideally, only be done for a few Entities, if at all.)
